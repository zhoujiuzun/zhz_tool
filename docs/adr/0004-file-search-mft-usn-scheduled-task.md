# 4. 文件搜索:MFT+USN 索引 + 计划任务提权 + 按搜索窗启停

状态:已接受(2026-06-20)

## 背景

新增「文件搜索」工具,对标 Everything 的体验:按文件名/文件夹名/路径近乎即时搜出本机文件。Everything 快的根因是直读 NTFS 的 **MFT(主文件表)**全量建索引 + 监控 **USN 变更日志**实时增量,而非逐目录遍历。但这两项都是 NTFS 特权操作,需管理员权限、仅对 NTFS 卷有效。

本程序原是普通权限的托盘工具(HKCU\Run 自启,纯 ctypes 调 Win32,单文件 exe)。引入 MFT/USN 与这个现状有根本冲突,需就"是否提权、如何提权、索引如何存取、如何保持最新"做一系列权衡。

## 决策

1. **走 MFT+USN,不走目录遍历**:接受"仅 NTFS、需提权"的代价,换取 Everything 级搜索体验。非 NTFS 卷(U盘/网络盘/FAT32)不纳入(暂)。
2. **提权方式 = 计划任务(最高权限 + 登录触发)**:首次设置创建计划任务(弹一次 UAC);之后 `schtasks /run` 静默拉起提权进程,**不再反复弹 UAC**。不做"零授权静默提权"(那是提权漏洞,且杀软会报)。
3. **搜索边界 = 名字+路径,不含内容**:只匹配文件名/文件夹名/完整路径,不做内容全文检索(全文是另一量级工程,与"快"冲突)。
4. **范围 = 全部本地 NTFS 盘,设置可勾选排除**;移动盘默认不纳入。
5. **索引 = 内存搜索 + 本地二进制存档**:首次读 MFT 全量建索引并落盘;重启从存档秒载,不重扫。搜索查内存(子串匹配 + 边打字边在上次结果里缩小)。**纯 Python 起步,在"存储+匹配"层留 C 替换接口**,实测不够快(目标盘约 4TB,可能上百万文件)再把匹配热路径换成 ctypes 调的小 C 模块。
6. **保持最新 = 按搜索窗启停的 USN 增量**:仅在搜索窗打开时监控 USN(打开→从上次 USN 位置补课→实时监控;关闭→停监控+落盘+退出提权进程)。窗口关闭即零后台(零进程、零内存、零监控)。

## 否决的备选

- **目录遍历 + 索引缓存(无需提权)**:搜索一样快、零权限、支持非 NTFS,但首次扫盘慢、实时性弱。否决因用户明确要对标 Everything 的扫盘速度与实时性。
- **Windows 服务模型**:Everything 官方推荐,但安装/IPC/生命周期复杂度远超托盘工具应有的量级。否决。
- **整程序常驻管理员**:每次启动弹 UAC 或需绕 UAC 的 hack,且 GUI 带管理员权限(安全面大、破坏拖拽)。否决,改为独立提权进程 + 计划任务。
- **24 小时常驻实时监控**:体验最实时,但没人搜时白占后台。否决,改为按搜索窗启停。

## 后果

- **好**:搜索体验贴近 Everything(秒级、改动实时);本地存档使重启不重扫;按搜索窗启停使不用时零后台;提权仅一生一次 UAC,日常不打扰;纯 ctypes 调 Win32,延续项目"零编译依赖"风格(C 加速是可选的后续升级,留好接口即可)。
- **代价 / 风险**:
  - **首次必弹一次 UAC**(创建计划任务),无法消除——这是合法性底线。
  - **计划任务"最高权限自启/按需拉起"模式,部分杀软会敏感**,可能误报。
  - **仅 NTFS**:非 NTFS 卷不支持(边界,非 bug)。
  - **USN 日志环形覆盖**:搜索窗长期关闭 + 磁盘海量改动,可能使补课失败 → 那次重开需全量重扫(罕见,自动兜底)。
  - **提权进程退出/拉起有几秒延迟**:开搜索窗时"加载存档 + USN 补课",需加载提示。
  - **与现有 `autostart.py`(HKCU\Run)的关系需理顺**:文件搜索的计划任务自启与主程序自启是两套,避免冲突/重复。
  - **C 加速若启用**:打破单文件纯 Python 打包的简单性(需编译器 + PyInstaller 打包 .dll/.pyd),故仅在实测确有必要时才引入。
- **约束**:MFT/USN 读取与计划任务操作必须在提权进程内进行;GUI 进程保持普通权限,两进程间需轻量 IPC 传递搜索请求与结果。

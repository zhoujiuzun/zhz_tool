# 2. 置顶窗按置顶时间严格排序,持续重申 z-order

状态:已接受(2026-06-19)

## 背景

窗口置顶工具可同时钉多个外部窗口。但 Windows 把所有 `WS_EX_TOPMOST` 窗口放进同一个 "topmost band",**band 内部谁压谁由"最后被激活(点击)的窗口"决定**——点哪个置顶窗,它就冒到这一档的最上。用户因此感觉"只能置顶一个"(其实都置顶了,只是新点的盖住了旧的),并要求:多个置顶窗之间要有**可控的优先级**,且**后置顶的优先级更高**。

核心矛盾:用户想要"固定的上下层关系"(后钉者恒在上),而操作系统的默认行为是"点谁谁冒头",两者直接冲突。

## 决策

**优先级 = 置顶时间(后钉者高),绝对严格,由程序持续重申 z-order 维持。**

- 维护一个按置顶时间排序的置顶窗序列,最新置顶的排最上。
- **绝对严格**:点击一个较早置顶的窗,**不**让它冒到更晚置顶的窗之上——程序持续(监听前台变化 / 定时)用 `SetWindowPos(..., SWP_NOACTIVATE)` 按序把每个置顶窗重新压回应在的层级。
- **改变顺序的唯一方式 = 重新置顶("提到最上")**:对某个置顶窗再置顶一次(先取消再置顶,即"双按热键"),刷新其置顶时间→升到最上。**不提供手动拖动排序,也不新增管理面板。**
- 够到被压住的较早置顶窗:靠用户**移走或最小化**压在它上面的窗(无需程序介入)。

## 否决的备选

1. **前台让步**(点哪个置顶窗它就临时冒到最上,松开焦点后其余按优先级恢复):更顺手、不和操作冲突,但**不满足"严格"语义**——用户明确要"点旧窗也不能冒头"。已否决。
2. **手动拖动排序 + 置顶管理面板**:可精细排序并解决"够不着旧窗"的问题,但需新增一个常驻小窗。被"移走/最小化压在上面的窗即可够着"这一点推翻,面板成了多余。已否决。

## 后果

- **好**:模型简单可预测(顺序只随置顶时间走,无需记忆谁排第几);零新增界面、零新增热键(复用现有 toggle);满足用户"后钉者恒在上"的诉求。
- **代价**:
  - **故意逆操作系统行为**——需持续 `SetWindowPos` 把窗口踩回时间序。未来维护者看到"点置顶窗却不让它上来 / 不停重排 z-order",极易误判为 bug 而拆掉本设计;本 ADR 即为防此。
  - 可能与其他 always-on-top 程序(PowerToys 等)互相抢 z-order;持续重压有**闪烁**风险,需控制重申频率。
  - **重叠的大窗场景下,点下层置顶窗会被上层置顶窗盖住**(因为严格维持时间序),此时下层窗虽获焦点却不可见——已知且用户接受的取舍。
- **约束**:z-order 重申须在主 GUI 线程进行(与 ADR-0001 同,Win32 窗口操作的线程约束)。

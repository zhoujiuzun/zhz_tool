# 5. 文件搜索:可切换搜索引擎(自研 + 委托 Everything)

状态:已接受(2026-06-21)

修订关系:**部分修订 ADR-0004**。0004 否决了"目录遍历等 MFT 之外的方案",确立 MFT+USN 为唯一数据来源。本 ADR 不推翻 0004(自研引擎仍是 MFT+USN),而是在其**之上**引入「可切换引擎」抽象,允许把搜索委托给外部 Everything,作为自研引擎成熟前的**过渡**备选。

## 背景

自研引擎(ADR-0004 的 MFT+USN 实现)已能用,但功能仍不完善(高级条件覆盖、子串匹配、多盘等仍在补齐)。用户希望在过渡期能切到成熟的 **Everything** 上,等自研引擎补齐后再切回。需求是:**同一套搜索窗 / 三种搜索模式 / 高级条件之下,底层引擎可在「搜索设置」里切换**。

这与 0004 的隐含前提冲突:0004 假设只有一个引擎(自研),整套「计划任务提权 + 按搜索窗启停 USN + 本地存档」都为它而生。引入第二个引擎,且该引擎是**外部产品**(可能未安装),需就依赖模型、默认策略、生命周期做权衡。

## 决策

1. **引擎抽象 = 上层契约不变,底层可换**:搜索窗只依赖一组方法(search / advanced_search / drives / len / ping),两个引擎各自实现。切换引擎**不改变搜索语义**(名字/路径/不含内容、三模式、高级条件、结果呈现一致),只改变"由谁来搜"。见 docs/CONTEXT.md「搜索引擎」。

2. **Everything 引擎 = 硬依赖运行中的 Everything + 检测回退;只捆绑 SDK 桩、不捆绑 Everything**:搜索委托用户本机**已安装且正在运行**的 Everything,经其 **SDK DLL(`Everything64.dll`,约 88KB 的 IPC 桩)** 通信。该桩**仅是进程间通信封装,Everything 没运行时它什么也查不到**——故"捆绑这个桩"与"分发 Everything 引擎本体"是两回事。运行时**优先探测用户 Everything 安装目录里的 DLL**,找不到才用捆绑副本(版本兜底)。检测不到 Everything 进程在运行时,搜索设置中该选项**置灰并提示「需先安装并运行 Everything」**,并**自动回退到自研引擎**。**不捆绑、不分发 Everything 程序本体**(避免体积膨胀、分发授权问题,保住 0004 的"单文件纯 ctypes、零编译依赖"基线;88KB 桩不破坏该基线)。

3. **默认引擎 = 记住上次选择;首次运行检测到 Everything 则默认它**:契合"过渡"意图(过渡期偏好成熟引擎),又不绑架用户——手动切换后该选择持久化(config)。自研引擎成熟后,改掉首次默认即可退役"过渡"定位。

4. **生命周期 = 按当前引擎决定 helper 启停**:自研那套「计划任务提权 helper + MFT 扫描 + USN 监控」**仅在自研引擎为当前引擎时启动**。选 Everything 时零提权、零计划任务、零 MFT 扫描——延续 0004「不用时零后台」原则,而非违反它。代价是中途切到自研有几秒"加载存档+USN补课"延迟(0004 已记录并接受的延迟)。

## 否决的备选

- **随程序打包 Everything(dll 或 ES + 服务)**:开箱即用,但 exe 体积大增、需处理 Everything 分发授权、要管理其服务生命周期,且与 0004「单文件纯 ctypes、零编译依赖」基线冲突。否决。
- **始终默认自研**:行为最可预期、不依赖外部,但与"自研不完善、要 Everything 过渡"的现实意图相悖——用户每次都得手动切到 Everything。否决,改为"首次检测到即默认 Everything + 记住选择"。
- **两引擎都常备(开搜索窗即拉起自研 helper)**:切换零延迟,但 Everything 用户被无谓提权、白扫 MFT、白占内存,违背 0004「零后台」。否决,改为按需启动。
- **选 Everything 即永久禁用自研(不能即时切回)**:最省,但搜索设置里的引擎切换无法即时生效,体验割裂。否决。

## 后果

- **好**:过渡期可用成熟的 Everything,自研不完善不阻塞使用;引擎抽象为将来加引擎(如非 NTFS 卷的目录遍历引擎)留好了seam;不打包外部产物,保住零依赖基线与小体积;按当前引擎启停 helper,Everything 用户零提权零后台。
- **代价 / 风险**:
  - **Everything 引擎依赖用户自行安装并运行**:未装/未运行则该选项不可用(边界,非 bug),回退自研。
  - **功能对齐的持续负担**:三种搜索模式 + 高级条件需在两个引擎上都映射到位(自研走 FileIndex、Everything 走其查询语法),否则"切换引擎不改变搜什么"的契约会破。这是引入第二引擎的长期维护税。
  - **结果细微差异**:两引擎对同一查询的命中集可能不完全一致(如 Everything 把符号链接/junction 别名也算入,自研按真实路径去重)。呈现一致,但**计数可能不同**,需在 UI/文档上对用户诚实(不声称两引擎结果逐条相同)。
  - **"过渡"定位需要退役动作**:自研成熟后,要主动改掉首次默认、并考虑是否下线 Everything 引擎,否则"过渡"会变成永久双轨。

## 约束

- 引擎抽象的契约(search / advanced_search / drives / len / ping)是两引擎的共同接口,新增引擎须实现它,且遵守"不改变搜索语义、结果呈现一致"的不变量。
- Everything 引擎只读 Everything 的查询结果,**不**触碰本项目的 MFT/USN/计划任务/存档。
- 引擎选择持久化在 config;切换须在搜索设置内完成,且尽量即时生效(开着搜索窗切换即换后端)。
